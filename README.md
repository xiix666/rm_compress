# [东南大学] 低码率吊射图传自定义客户端开源文档

本项目基于 CompressAI 的 `mbt2018_mean` 预训练模型和 QVRF 可变码率控制，实现低码率吊射图传。当前 QVRF 分发包支持 `192 x 192 @ 24 FPS` 实时传输：链路侧为 48 Hz、每帧两个 300 B 物理包；协议头占用后，实际图像 payload 预算约为 `2 x 280 B = 560 B`。接收端可使用 fused TensorRT `g_s + RLFN x2` 路径，输出 384 x 384 画面。

## 项目特点

1. 安全可控：使用 QVRF 实现可变码率控制，在两个 300 B 物理包的链路限制下，将实际图像 payload 控制在约 560 B 内。
2. 推理高效：发送端能在 12 ms 内完成大部分帧的推理压缩；少量超预算帧可以在下一帧到来前完成重试。接收端使用 TensorRT fused `g_s + RLFN` 路径完成重建与超分。
3. 鲁棒性强：基于预训练模型，不依赖强场景先验规则，场上更难遇到特定规则失效的 corner case。
4. 功耗较低：操作手实测笔记本无外置电源可以在场上打三场；在充电宝帮助下预计可以打五场。
5. 方便部署：可以在 agent 帮助下快速部署到多台自定义客户端电脑。本代码基于 RTX 4060 Laptop 开发，经测试，50 系显卡可通过简单修改并重建 TensorRT engine 完成部署。同时客户端已经完成了与裁判系统和官方客户端的通信，部署成功后可以直接使用。

## 方案选型

方案选型时重点考虑三个问题：

1. 压缩方案是否能严格满足低预算；如果部分帧超预算，是否可以补救。
2. 方案是否足够稳定。由于手头没有场上内录，为了避免上场出现问题，方案必须足够鲁棒。
3. 画质和帧率是否能满足操作手吊射需求。

前期测试发现：

1. 操作手对画质要求不高，即使压缩到 128 x 128，仍然能够判断落点。
2. 操作手对帧率要求更高，但 24 FPS 已经足够使用。Bilibili 上的第一人称视角吊射视频其实是 25 FPS，操作手反馈可以看得清落点。

调研方向主要有三类：

1. 传统压缩方案：逐帧独立压缩很难达到预算；H.264 等帧间压缩虽然平均码率可以很低，但关键帧仍然容易超预算，因此没有采用。
2. 基于先验的方案：例如利用吊射时背景不变、只有弹丸运动来筛选关键信息。但我们的英雄吊射时抖动较大，静态背景提取困难；在当时时间不足的情况下，这条路线风险较高。
3. 深度学习压缩方案：CompressAI 已经内置多种成熟模型。简单测试后发现 `mbt2018_mean`（MS-SSIM 目标）能让大部分帧落在 560 B 预算内，并且可以接入变码率控制算法，因此最终沿这条路线发展。

上述调研并不代表其他方案一定做不出来，只是当时开发时间有限，没有继续深入尝试。

最终方案：

- 算法：CompressAI `mbt2018_mean` + QVRF 码率控制；接收端使用 fused TensorRT `g_s + RLFN x2` 做重建与超分。
- 推理后端：OpenVINO 与 TensorRT。发送端 `g_a` 可用 iGPU + OpenVINO；接收端 fused `g_s + RLFN` 使用 TensorRT。
- 机载发送端硬件：海康 CS200-UC（调低分辨率可以 24 FPS）+ 50 mm 镜头 + NUC12WSKi5。（用usb转ttl发送比特流给图传串口）
- 自定义客户端硬件：NVIDIA GPU 笔记本电脑。本代码基于 RTX 4060 Laptop，经过简单适配并重建 TensorRT engine 后，可以部署到 50 系显卡笔记本。

## 具体实现

### 原理简述

`mbt2018_mean` 的压缩流程比较复杂，这里只做直觉性说明，具体方法需要看论文和 CompressAI 实现。

原始图像 `x` 会经过神经网络 `g_a` 压缩为 latent 表示 `y`。`y` 承载主要可重建信息，元素数量比原图少很多，但从 bit 数看仍然较大，因此还需要统计建模和熵编码。

模型假定 `y` 的元素服从高斯分布，可以用 `mean` 和 `scale` 表示。`h_a + h_s` 的作用是预测 `y` 中每个元素对应的 `mean` 和 `scale`：`y` 先经过 `h_a` 得到更低维的统计信息 `z`，`z` 再经过 `h_s` 得到 `means_hat / scales_hat`。

有了 `y`、`means_hat` 和 `scales_hat` 后，就可以进一步熵编码。QVRF 中可直观理解为：

```text
symbols = round(gain * (y - means_hat))
```

`gain` 越小，量化步长越大，更多相近数值会被量化到一起，symbol 分布更集中，也就更容易压缩。

`scale` 表示分布宽度，决定某个 symbol 使用哪张概率表。实际实现使用 CDF（Cumulative Distribution Function，累计分布函数）表。例如某个 scale 下：

```text
symbol    概率
-2        0.01
-1        0.10
 0        0.78
 1        0.10
 2        0.01
```

对应的 CDF 区间可以理解为：

```text
symbol    CDF 区间
-2        [0.00, 0.01)
-1        [0.01, 0.11)
 0        [0.11, 0.89)
 1        [0.89, 0.99)
 2        [0.99, 1.00)
```

这个表建立了 symbol 与 CDF 区间的映射。比如编码 `symbols = [0, 1, 0]`：

1. 第一个 symbol 是 `0`，区间为 `[0.11, 0.89)`。
2. 第二个 symbol 是 `1`，在当前区间内部取 89% 到 99% 这一段，得到新区间 `[0.8042, 0.8822)`。
3. 第三个 symbol 继续更新，得到 `[0.81278, 0.87362)`。

只要选一个落在最终区间里的数，就可以表示整串 symbols。实际实现不会使用小数概率，而是使用整数 CDF 和 rANS 编码。

因此直观理解是：传输 `y_string`、`z_string` 以及约定好的 CDF 和解码规则，就可以还原 `y`，再通过 `g_s` 重建图像。

为什么需要传输 `z_string`？因为 `mean` 和 `scale` 本身与 `y` 形状对应，如果直接传输会很大。实际流程是：

```text
编码端：y -> h_a -> z -> 熵编码 -> z_string
解码端：z_string -> 熵解码 -> z_hat -> h_s -> mean / scale
```

编码端也必须计算一遍 `h_s`，因为它需要用同样的统计信息压缩 `y`。因此，编码端和解码端的 `h_s` 输出一致性非常关键。

### 码率控制

QVRF 采用增益缩放进行码率控制。在编码隐变量时，通过乘以增益系数 `gain` 来压缩特征幅度，从而减少熵编码 bit 数，而无需修改模型权重。

直观理解：`gain` 越小，量化步长越大，编码内容更容易量化成相同数字。例如量化步长为 1（`gain=1`）时，`[0.3, 0.7, 1.6]` 会被量化成 `[0, 1, 2]`；如果量化步长为 2（`gain=0.5`），则会被量化成 `[0, 0, 1]`，反量化后约为 `[0, 0, 2]`。此时数字分布更集中，因此更容易压缩。

发送端采用自适应反馈机制预测下一帧增益。根据前一帧的码流填充比例（实际字节数 / 560 B payload 预算）自动调整 `gain`：

- 填充率低于 72% 时，提高 `gain` 以改善质量。
- 填充率高于 94% 时，降低 `gain` 以压缩码率。
- 超预算时，更激进地降低 `gain`。

若一次编码超预算，会启动重试机制：第二次尝试降低 `gain` 重新编码，第三次继续激进降低。如果仍然超预算，发送 `OVER` 标记，接收端保留前一帧显示。实测大部分帧可以第一次编码成功；由于平均压缩延迟约 10-12 ms，即使少量帧需要重试，也可以满足 24 FPS 的实时要求。

### 推理与优化

这种学习压缩算法的推理优化不能完全套用一般视觉模型，需要考虑熵编码的一致性。

为了压缩和解压 `y`，编码端与解码端都需要 `mean / scale`。这些统计信息由 `h_s` 产生，因此 `h_a / h_s` 必须在发送端和接收端使用一致的推理后端与设备配置。当前可用方案包括两端均使用 CPU + OpenVINO，或两端均使用相同型号 iGPU + OpenVINO。实测在两端一致时可以保证正确；如果混用 iGPU 与 NVIDIA GPU 等异构组合，可能因数值偏差导致解码乱码。

剩下的优化相对常规：

- 发送端使用 iGPU + OpenVINO 加速 `g_a`。
- 使用 OpenVINO PrePostProcessor 将 BGR uint8、resize、RGB 转换、归一化融合到推理图中，避免 CPU 预处理。
- 接收端用 TensorRT 加速 `g_s` 和 RLFN。由于 `g_s` 和 RLFN 联系紧密，中间没有必须回到 CPU 的步骤，因此将两个模型融合为一个 TensorRT engine，直接输出超分后的图像。
- 后处理部分使用 CuPy 替代 NumPy，把 `NCHW FP32 -> HWC uint8` 转换放到 GPU 上，提高速度。
- 将 CompressAI 的 `build_indexes` 改成 NumPy `searchsorted`，减少搜索开销。

这些优化除了提高速度，也降低了功耗。例如将 PyTorch CUDA 换成融合后的 TensorRT 模型后，显卡功率明显降低，从而提高自定义客户端续航能力。

## 实测结果

发送端：

- 压缩总耗时 avg：11.06 ms/frame
- p50：10.94 ms
- p90：11.26 ms
- p99：13.14 ms
- max：29.37 ms（首帧 warmup + 2-pass，非稳态）
- 0 over budget，最大 packed 559 B

接收端：

- GPU：RTX 4060 Laptop GPU
- 测试视频：`dataset/raw_camera/Video_20260505120921084.avi`
- 接收端路径：QVRF 192，TensorRT fused `g_s + RLFN SR`
- receiver 吞吐：168.43 FPS
- decode latency：avg 5.92 ms，p50 5.85 ms，p90 6.28 ms，p99 6.96 ms

功耗：

- idle baseline：avg 16.91 W
- receiver running：avg 46.94 W，p50 50.19 W，p90 50.75 W，max 51.78 W
- 净增加功耗：约 30.03 W

以上优化比较粗糙，但当时的目标是“够用就行”，因此没有继续做更细致的工程化优化。例如后处理可以继续改成手写 CUDA，接收端也可以进一步改成 C++ 实现，理论上还能更快。
